## 현재 한계

기존 텔레그램 봇은 이슈/상태를 물리적으로 체크해서 알림 발사:

* "PACAA-XX 승인 대기"
* "PACAA-YY 검토 대기"

문제: 보드(나)가 **무엇을 해야 하는지**가 불명확. 본문 보고
이슈 들어가서 컨텍스트 다시 파악해야 함. 시간 낭비.

## 새 구조

작업이 보드의 액션/판단/승인/선택 없이 재개 불가능한 시점이면
즉시 텔레그램 발사. 본문 형식은 다음 3가지로 분리:

### A. 액션 알림 (보드가 직접 무언가 해야 하는 경우)

본문 구성:

* 한 줄: 보드가 무엇을 해야 하는지 (예: "Vercel 결제 카드 등록")
* 왜 필요한지 한 줄 (예: "Plausible 셋업 위해 결제 카드 필요")
* 첨부: 근거 텍스트 파일 (모바일에서 열람 가능)
  * 상세 컨텍스트
  * 이 액션이 진행되지 않으면 어떤 작업이 막히는지
  * 진행 후 다음 단계

### B. 승인 요청 (보드가 승인/거절만 하면 되는 경우)

본문 구성:

* 한 줄: 무엇을 승인하는지
* 승인 시 영향 한 줄
* 거절 시 영향 한 줄
* 텔레그램 인라인 버튼: 승인 / 거절 (직접 응답 가능)

승인을 위해 보드가 알아야 할 최소 정보를 본문에 포함.
긴 컨텍스트는 첨부 파일.

### C. 선택지 요청 (보드가 여러 옵션 중 선택하는 경우)

본문 구성:

* 한 줄: 무엇을 결정하는지
* 모든 선택지 나열 (각 선택지 한 줄 설명)
* AI 추천 선택지 별도 표시 (예: ⭐ 표시)
* 추천 근거 한 줄
* 각 선택지의 영향 (간단히)

선택지를 누를 수 있는 인라인 버튼 또는 응답 형식 명시.
긴 컨텍스트는 첨부 파일.

## 가독성 원칙 (모든 텔레그램 출력에 적용)

기존 텔레그램 알림 + 능동 스캔 보고 + 일일 다이제스트 + 보류 트리거

* 신규 추가될 모든 출력에 다음 원칙 일괄 적용:

1. **한국어 우선**. 영어 약어/시스템 용어 최소화.
2. **일상 용어**. "임계값", "트리거", "deferred", "stale", "DoD" 같은
   페이퍼클립/기술 용어 대신 평이한 단어:
   * "임계값 도달" → "이 숫자에 도달함"
   * "트리거 발동" → "이 조건이 충족됨"
   * "deferred 항목" → "보류해뒀던 일"
   * "stale issue" → "오래 멈춰있는 작업"
3. **티켓 번호 최소화**. 본문에 PACAA-XX 같은 번호

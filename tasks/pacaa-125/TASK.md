## 현재 한계

PACAA-117 승인 요청 알림 받음. 가독성 여전히 부족.

문제:

* "PACAA-117" 같은 시스템 코드부터 시작 (호명 없음)
* "child B", "lib probe", "SELECT/LIMIT", "generator hard cap" 같은
  기술 용어 그대로 노출
* "왜 승인해야 하는지" 큰 그림 부족
* "승인 후 무엇이 진행되는지" 명확하지 않음
* 직전 대화 인용 잘림

## 새 형식 표준

모든 승인/액션/선택 텔레그램 알림에 다음 5단계 구조 강제:

### 1. 호명

첫 줄에 "김선혁님," 으로 시작. 사람 대 사람 톤.

### 2. 무엇 (현재 상황)

지금 무엇이 발생했는지 일상 용어로 한 단락. 시스템 코드 / 약어 /
기술 용어 금지.

예시 변환:

* "sitemap 1,031 URL, 실제 vendor ≥1,200"
  → "사이트맵에 1,031개 vendor만 등록됨. 실제로는 1,200개 이상
  등록되어야 함. 약 200개 누락."

### 3. 왜 (배경/맥락)

이게 왜 중요한지, 본인 회사 비전/Goal에 어떻게 영향 주는지.

예시:

* "누락된 vendor는 검색엔진 노출 안 됨"
* "SG-2 SEO 점유율 달성에 영향"

### 4. 작업 내용 (승인 후 진행)

승인하면 어떤 작업이 진행되는지 단계별로 평이한 한국어로.
기술 디테일은 첨부 파일에. 본문은 일상 용어.

### 5. 김선혁님이 결정할 것

명확한 선택지. 승인하면 즉시 어떤 결과가 발생하는지 한 줄.

## 일상 용어 강제

다음 시스템 용어 → 일상 한국어 매핑 (확장 가능):

| 시스템 용어       | 일상 한국어                   |
| ------------ | ------------------------ |
| child issue  | 하위 작업                    |
| sitemap      | 사이트맵 (검색엔진에 보내는 페이지 목록)  |
| probe        | 검증                       |
| hard cap     | 제한                       |
| generator    | 자동 생성 시스템                |
| SELECT/LIMIT | (DB 코드, 본문 노출 금지 → 첨부에만) |
| preview      | 시험 환경

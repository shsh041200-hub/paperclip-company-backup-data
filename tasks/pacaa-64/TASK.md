## 1. 저장 시 명확화 의무

보드가 보류/부분 채택/조건부 결정을 내릴 때 CEO는 다음 둘 중
하나로 처리:

A. 시기/조건이 명확한 경우: 즉시 deferred\_items.md에 저장
B. 시기/조건이 명확하지 않은 경우: 저장 전 보드에게 질문

* "이 항목 재검토 트리거를 어떻게 잡을까요?"
* 보드가 답 못 주면 CEO가 합리적 추천 후 보드 확인
* 트리거 확정 후 저장

트리거 없이 저장 금지.

## 2. 저장 작업의 운영 방식

보류 항목 저장 작업은 다음 규칙으로 진행:

* 별도 이슈 생성 (저장 자체를 작업으로 등록)
* 우선순위: 최하위
* Goal/Vision 직접 기여 아님
* 사이드 프로젝트 무게 (분해 최소화, 가벼운 보고)
* 본 작업(SG-1, SG-3 등) 진행에 영향 금지

## 3. 트리거 도래 시 알림 흐름 (수동 트리거)

트리거 도래 감지 시:

1. CEO가 보드에게 텔레그램 봇(Paperclip0412\_bot 활용)으로 보고
2. 보고 형식:
   * 보류 항목 한 줄 요약
   * 실행 적절한 시기인 근거
   * CEO 판단 한 줄 (다음 중 하나, 자가 판단):
     * "지금 도입 권장" + 짧은 근거
     * "조금 더 미루기 권장" + 짧은 근거
     * "도입 또는 폐기 갈림길" + 짧은 근거
   * 보드의 결정 옵션 (실행 / 더 미루기 / 폐기)
3. 보드가 텔레그램 또는 Paperclip 코멘트로 응답
4. 승인 시 실행 진행
5. 거절/추가 보류 시 재트리거 조건 받아서 재저장

CEO 판단 한 줄 작성 원칙:

* 한 문장 이내
* 보드의 5초 결정에 결정적 정보만
* 의견을 명시 ("권장합니다" / "비권장입니다")
* 근거는 짧게 (예: "페어 1이 아직 첫 출하 단계라 부담")

## 4. 기회 감지 (능동 스캔)

수동 트리거 외에 능동 스캔도 수행. 매 하트비트 Phase 6 중:

deferred\_items.md의 모든 보류 항목에 대해 다음 자가 질문:

1. 조기 충족 점검: 이 항목의 트리거가 공식적으로 안 됐지만,
   실질적으로 충족된 상태는 아닌가?
   예: 트리거 "페어 1 안정 후" → 정식 2주 검증 전이지만
   실제 출하 패턴이 안정적이면 충족 가능.
2. 상황 변화 점검: 외부 환경 또는 내부 상태 변화로 이 항목이
   "지금 도입하면 가치가 더 큰" 상황은 아닌가?
   예: 보

## 발견한 근본 빈 구멍

현재 회사의 거의 모든 작업이 시간 기반 마감으로 정의됨:

* "3일 안에 완료"
* "1주일 후 시작"
* "5월 19일까지"

이 패턴이 인간 회사에서는 효율적이지만, AI 에이전트 회사에서는
시간 낭비를 초래.

### 구체 손실

* 작업이 30분 만에 끝나도 시스템이 3일 마감으로 인식 → 다음
  작업이 마감일까지 대기
* 체인 효과: A → B → C 작업이 실제 1.5시간이면 끝나는데
  1주일 걸림
* 에이전트 본연의 강점(매 하트비트 활성화, 병렬 처리, 즉시
  실행) 활용 안 됨

이 패턴이 plan v3 사인오프 후 모든 하위 작업에 박혀있음. 즉시
교정 필요.

## 새 운영 원칙: Event-driven Orchestration

다음 원칙을 회사 운영의 근본 철학으로 박을 것.

### 원칙 1: 작업 마감을 두 카테고리로 분리

**카테고리 A: 시간 마감 정당 (Time-based justified)**

* 외부 의존성 (예: SEO 인덱싱 12주, Vercel 응답 36시간)
* 데이터 축적 필요 (예: 1개월 사용자 행동 분석)
* 외부 약속/이벤트 (예: 분기 보고)
* 시장 자연 사이클 (예: 한국 패키징 성수기)

이 카테고리는 시간 마감 유지.

**카테고리 B: 이벤트 마감 정당 (Event-based justified)**

* 제작/생산 작업 (코드, 디자인, 콘텐츠)
* 검증 작업
* 의사결정 작업
* 학습/회고 작업

이 카테고리는 시간 마감 제거. "이전 작업 완료 + 검증 통과 시 즉시"
형태로 정의.

### 원칙 2: 작업 정의 시 자가 질문 의무

모든 에이전트가 새 작업/이슈 생성 시 자가 질문:

"이 작업의 마감이 진짜 시간에 의존하는가,
아니면 이전 작업 완료에 의존하는가?"

후자면 시간 마감 박지 말 것. 트리거 조건으로 정의.

### 원칙 3: 트리거 체인 자동화

이벤트 기반 작업 정의 시:

* 선행 조건 명시 (어떤 이벤트가 발생하면)
* 즉시 실행 (마감일 대기 X)
* 후행 작업 자동 알림 (이 작업 완료 시 다음 작업 자동 트리거)

### 원칙 4: 시간 마감의 의미 재정의

시간 마감이 정당한 카테고리 A에서도:

* 마감일 \= 데드라인 (그때까지 끝내야 함)이 아닌
* 마감일 \= 검토 시점 (그때 진척 확인)으로 재정의
* 작업이 일찍 끝나면 즉시 다음 단계 진행
* 작업이 마감일까지
